Basically, the problem is this:

On boot, DSLinux looks to see if a /etc/passwd file exists and if it doesn't, it generates one. The reason why the GnM fails is because for some reason with later versions of DSLinux it only works read only. This means that the /etc/passwd file doesn't get generated which a lot of Linux programs rely on, which causes all the errors.

What people have been doing with the GnM is supplying their own /etc/passwd file to skip the part where it tries to generate one. This can hardly be called a fix as the entire root filesystem will be readonly and various Linux applications (like gpm for example) need to be able to write PID files and the like.

As I've said over on the DSLinux forums, I am no longer interested in making DSLinux work on Slot-1 devices as they cannot provide extra RAM which DSLinux so sorely needs.

See here for a list of known working Slot-2 RAM packs.
